# Огляд проекту dzenua (dzen)

Короткий аналіз реальної структури та запуску. Робочий код знаходиться в **`web-app/`** (моноліт: Express + React-клієнт).

---

## 1. База даних: чи є, чи потрібна, чи використовується

**База є і використовується.**

- **СУБД:** MySQL (драйвер `mysql2`). У `config/database.js` задані параметри підключення (за замовчуванням — віддалений хост `rv596484.mysql.tools`; можна перевизначити через змінні середовища).
- **Таблиці (створюються при старті):**
  - **`media`** — використовується: список відео/аудіо/зображень, завантаження, оновлення, видалення через API.
  - **`admin_users`** — створюється, але **не використовується**: вхід в адмінку йде через HTTP Basic Auth з логіном/паролем із змінних середовища (або дефолтів у коді).
  - **`settings`** — створюється, але **не використовується**: налаштування інтерфейсу (фон, валюта, музика тощо) зберігаються тільки в **localStorage** на клієнті.

**Висновок:** База потрібна для медіа (список і метадані). Без MySQL сервер не зможе показувати/додавати медіа через API; при помилці підключення API медіа повертає захардкожений приклад даних.

---

## 2. Адмінка

**Так, адмінка є.**

- **Де:** у тому ж React-додатку.
  - Сторінка: **`/admin`** (роут у `client/src/main.tsx` → `AdminApp.tsx`).
  - Модальна панель на головній: кнопка викликає `AdminPanel.tsx` (завантаження/перейменування/видалення медіа).
- **Вхід:** HTTP Basic Auth. Логін/пароль беруться з `process.env.ADMIN_USERNAME` та `process.env.ADMIN_PASSWORD`, за замовчуванням у коді: `admin` / `zenua_admin_2025`.
- **Можливості:** перегляд списку медіа, завантаження файлів (тип: відео/аудіо/зображення), перейменування, видалення.

Окремої адмін-збірки (як у README для `admin/`) у поточному репозиторії немає — все в межах `web-app`.

---

## 3. Куди завантажувати відео, музику, картинки і в яких форматах

**Куди:**

- Файли зберігаються на сервері в каталозі **`web-app/uploads/`** у підпапках:
  - **`uploads/video/`** — відео  
  - **`uploads/audio/`** — аудіо  
  - **`uploads/image/`** — зображення  
- Тип визначається полем форми при завантаженні (`type`: `video` | `audio` | `image`). До клієнта медіа віддаються по URL виду `/uploads/video|audio|image/ім'я_файлу`.

**Форматы:**

- У коді **немає перевірки MIME/розширення**: multer приймає будь-який файл. У кореневому `env.example` вказані `ALLOWED_FILE_TYPES` та `MAX_FILE_SIZE`, але **у `web-app/server.js` вони не використовуються**.
- Рекомендовано підтримувати типові формати:
  - **Відео:** MP4, WebM  
  - **Аудіо:** MP3, WAV, OGG  
  - **Зображення:** JPEG, PNG, WebP  

Щоб обмежити формати та розмір, потрібно додати в multer `fileFilter` і `limits`, використовуючи змінні з `.env`.

## 4. Як розмістити сайт на хостингу

Фактично запускається один Node-сервер із `web-app/` (Express), який:

- віддає API (`/api/*`);
- віддає статику з `dist/` (збірка React) і з `uploads/`, `storage/`.

**Кроки для продакшену:**

1. **MySQL:** створити базу на хостингу (або окремий сервіс типу Clever Cloud, PlanetScale тощо) і вказати в `.env` (див. нижче).
2. **Збірка клієнта:**  
   - У `web-app` зараз: `"build": "vite build"`. Vite збірка йде з `root: './client'`, результат за замовчуванням у `client/dist/`.  
   - Сервер очікує статику в `web-app/dist/`. Потрібно або змінити `vite.config.js` (`outDir: '../dist'`), або після `npm run build` копіювати вміст `client/dist/` у `dist/`.
3. **Запуск:** у папці `web-app` виконати `npm run build` (після налаштування шляху до статики), потім `node server.js` або `npm start`. Порт задається змінною `PORT` (за замовчуванням у коді — 8003).
4. **Проксі:** на хостингу (Nginx/Apache/Caddy) налаштувати проксі на один порт (наприклад, 8003), щоб до нього йшли запити до сайту та до `/api` і `/uploads`.
5. **Файли завантажень:** каталог `web-app/uploads/` має зберігатися між деплоями (volume або окремий диск), інакше завантажені медіа зникнуть після перезапуску/редеплою.

**Увага:** кореневий `docker-compose.yml` і README описують структуру з окремими `frontend/`, `backend/`, `admin/` і PostgreSQL — у репозиторії їх немає. Для поточного коду потрібен окремий Dockerfile під `web-app` (Node + MySQL), якщо хочете деплоїти в контейнерах.

## 5. Чи потрібно створювати .env

**Так, краще створити.**

- У `web-app` пакет **dotenv** є в залежностях, але у **`server.js` немає виклику `require('dotenv').config()`**, тому файл `.env` зараз не підхоплюється. Якщо додати на початку `server.js`:
  ```js
  require('dotenv').config();
  ```
  то змінні з `web-app/.env` будуть доступні в `process.env`.
- Що має бути в **`web-app/.env`** (можна взяти за основу кореневий `env.example`, але орієнтуватися на те, що реально читає код у `web-app`):

| Змінна | Призначення |
|--------|-------------|
| `PORT` | Порт сервера (за замовчуванням 8003) |
| `DB_HOST`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`, `DB_PORT` | Підключення до MySQL (зараз у `config/database.js` є дефолти, краще задати свої) |
| `ADMIN_USERNAME`, `ADMIN_PASSWORD` | Логін/пароль для Basic Auth адмінки (обов’язково змінити в проді) |

Кореневий `env.example` містить також PostgreSQL, Redis, AWS, SendGrid тощо — у поточному `web-app` вони не використовуються; їх можна ігнорувати, поки не додасте відповідний функціонал.

## Підсумок

| Питання | Відповідь |
|---------|-----------|
| База є і потрібна? | Так, MySQL; використовується для медіа. Таблиці `admin_users` і `settings` не використовуються. |
| Адмінка є? | Так: маршрут `/admin` і модальна панель на головній; вхід через Basic Auth. |
| Куди і в яких форматах завантажувати медіа? | У `web-app/uploads/{video,audio,image}/`; форматы в коді не обмежені — варто додати перевірку з `.env`. |
| Як викласти на хостинг? | Node (з `web-app`) + MySQL, зібрати клієнт у `dist`, проксувати трафік на один порт, зберігати `uploads`. |
| Чи потрібен .env? | Так; додати `dotenv.config()` у `server.js` і створити `web-app/.env` для PORT, DB_*, ADMIN_*. |
